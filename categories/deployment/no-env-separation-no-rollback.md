# Skipped dev/staging/prod separation — deployed vibe-coded changes with no rollback path

## What happened

Developed and deployed directly to production. The vibe coding workflow felt fast enough to "just fix it in prod." Made changes, pushed, deployed. When something broke, the fix was another vibe-coded change pushed directly to prod.

Eventually a bad deployment took down the service. The "fix" introduced a second bug. Rolling back wasn't straightforward because there was no clean prior state to roll back to — just a chain of hot-fixes on top of each other. Recovering required manually reconstructing a working state from git history.

## Root cause

Vibe coding creates a false sense of confidence. The speed of generating code makes it feel like the risk is low — "if it breaks, I'll just fix it in 2 minutes." But vibe-coded changes carry higher risk than hand-written changes, not lower, because:

1. You have less line-by-line understanding of what changed
2. The coding agent may have modified files you didn't expect
3. Side effects are harder to predict when you didn't write the code yourself

Without environment separation, every deployment is a bet with no hedge. Without rollback capability, every failed bet requires manual recovery.

## Fix / Lesson

Set up the same environment separation that existed before AI:

```
dev     → experiment freely, break things, test ideas
staging → integration testing, validate before prod
prod    → deploy only validated changes, rollback always available
```

Takeaways:

- **Vibe coding changes who writes the code, not the physics of deployment.** Services still crash, integrations still break, edge cases still exist. The safety nets exist because the problems exist.
- **Rollback capability is more important with vibe coding, not less.** You have less understanding of what changed — you need more ability to undo it.
- **"I'll just fix it" is not a rollback strategy.** A hot-fix under pressure, generated by a coding agent, has a high chance of introducing new issues. A clean rollback to known-good state is always safer.
- **Fast code generation does not mean low deployment risk.** Speed of writing and safety of deploying are unrelated. Don't let one create false confidence in the other.
